Toast

Toast floats over page content and displays low-priority messages in a minimally disruptive way. It is self-dismissing and does not require action.

Forge Extension

You must import this component separately.
It is not part of Forge Core yet. It may undergo further design/API changes. See the Extensions page for more info.

Usage

Toast appears for several seconds, displays a short message, and then disappears. It’s designed to catch attention without interrupting. It can also include a button for taking action. Users can manually dismiss a toast instantly, and they can hover to keep it visible (a helpful accessibility feature if someone needs more time for reading or uses assistive technology).

Toast is used to inform the user of low-priority events, such as:

  • Events outside the user’s current workflow (lab results, system updates)
  • Non-interruptive confirmations for actions in the current workflow (successful saves)

Do not use Toast for information that requires immediate action or attention. Toast is for low-priority messages where it's OK if the user misses them in that moment.

Use this scale to help decide your notification's level of interruption:

Toast uses Forge color semantics for its colors and icons.

Due to the design of Toast (visible for a short time and deliberately low on disruption), users might not read all of its content or may miss it completely. Therefore, do not put information in a toast that can't be accessed somewhere else. Consider building a repository for past toast messages if you're concerned about users needing to refer back them.

Visual Language and Design

Toast offers 3 message styles (using Forge color semantics and icons):

  • Error or warning (Attention icon)
  • Information (Information icon)
  • Success or Confirmation (Success icon)

These colors and icons communicate (at a general level) the message type in a way that users can quickly and easily understand. These icons were designed to provide a specific level of visual weight and emphasis. They cannot be replaced with other icons.

Toast includes a drop shadow and border to help separate it from the background.

Size

  • Fixed width: 375px
  • Height depends on the length of toast content; maximum is 105px
  • Width of the action button depends on its label text; maximum is 125px

Behavior

Animation and timing is built into Toast.

A toast remains fully visible for 7 seconds before self-dismissal happens. After that, the toast will fade in place over 3 seconds. Users can hover to make the toast fully visible again, giving them more time to read and respond.

The Dismiss button or icon is always displayed, allowing users to readily dismiss the toast. When the Dismiss icon/button is clicked, the toast instantly closes and disappears.

One action button can be included in the toast. Clicking the button closes the toast and carries out the action. When deciding what actions to offer, make sure that the toast message will display sufficient information for users. Allowing users to take actions without enough information could lead to errors and patient safety issues.

Position

Toasts are displayed on the bottom left-hand corner to minimize disruption and blocking of call-to-action buttons or critical information. This position also has close proximity to the Status bar on the bottom of the viewport.

Content

The content of toasts should be minimal and short. The recommended amounts are:

  • Header: 1-3 words, one row
  • Message body: 1-2 sentences or rows of information
  • Button: 1 word, one row

Design your content so that users can quickly scan it. Refer to the content style guide for advice on writing concisely and using plain language.

Consider the purpose of your content, what it needs to communicate to users, and what you want them to do after reading it. For example, a toast could:

  • Provide a very short, self-explanatory message that needs no action (e.g., the message "Your file was successfully uploaded")
  • Provide just enough information about an event that the user can decide whether to engage further now or leave it until later (e.g., a patient's name/date of birth/ID number and the words "Radiology report")

Character Truncation

Text in the message of the toast can be truncated if viewing the whole message is not necessary (for example, if the message is a preview of lab result data). In this case, the toast's action button should provide a way for the user to view the full message. However, consider whether it would be more helpful and clear to show a complete, deliberately written message instead.

Avoid truncating the header and button labels, for maximum clarity.

This truncation does not automatically add an ellipsis at the end of the truncated content. See the Implementation section for guidance on adding an indicator for truncated content.

Privacy and Protected Health Information

If your toast message displays Protected Health Information (PHI) or other sensitive information, consider whether it may be seen by someone who shouldn't have access to that information.

For example, in some clinical settings, providers and patients review the patient's chart together on the doctor's monitor during a visit. If a toast appeared during that time, the patient could accidentally see another patient's name, birthdate, results, or other private information in the toast message.

Multiple Toasts

Toasts are displayed one at a time.

A displayed toast will immediately self-dismiss and disappear if there is another toast following.

If multiple toasts are triggered at the same time or very close to one another, the most recently triggered toast will be displayed.

Accessibility

Toast includes several built-in features that support accessibility, including:

  • Keyboard shortcut (hotkey)
    • If the toast includes an action, users can enter the key combo Ctrl-Shift-1 to fire the action (without needing to use a mouse).
  • Screen reader support
    • Screen readers will announce the toast's title and message. If the toast includes an action, the screen reader will also announce "Press Ctrl-Shift-1 to (action label)." Use the labelForScreenReader prop if needed to provide an alternative action label for screen readers. For example, if the label is "Details", then a screen reader would announce "Press Ctrl-Shift-1 to Details", which is confusing. Here, labelForScreenReader could be set to "View Details", and the screen reader would announce "Press Ctrl-Shift-1 to View Details" instead.
    • ARIA attributes for screen reader support: aria-live: polite; aria-relevant: additions
  • Hover to maintain visibility
    • User can hover over a toast to keep it visible for as long as necessary. This is helpful for people who use assistive technology or who need more time for reading (e.g., due to dyslexia, cognitive issues, English as a second language).

Implementation

One Toast at a Time

Toast is displayed for just a short time and only one can display at a time, regardless of how many Toast instances are in the React render tree and for how long. This leads to several special considerations when using Toast:

  • id is required and should be unique for every new message. If you use the same id for different messages, you risk a new message not being displayed because it’s mistaken for a previously displayed message.
  • Rendering a new Toast will force the currently displayed Toast to dismiss immediately.
  • If several Toast instances are rendered in quick succession, some of them may not be displayed at all. This is acceptable because Toast shouldn’t be used for critical information and it's OK if the user doesn't see it. You can use the onNeverShown callback to detect when this has happened.

Message

Whatever is passed in as children will be rendered as the toast message, under the heading. This is only to support stylized and structured text. Do not insert images, other non-textual content, or interactive content into toasts.

Toast has intentionally limited space as described above so that it’s less disruptive to the workflow. If a message isn't fully visible, however, Toast does not show an ellipsis to indicate this. Teams can add this themselves if necessary (one of the uses of structured children). Keep in mind that truncating multi-line text is harder than truncating single-line text.

import Toast from '@athena/forge-toast/build/components/toast/toast';
import ToastTemplate from '@athena/forge-toast/build/components/toast-template/toast-template';
import ToastPortal from '@athena/forge-toast/build/components/toast-portal/toast-portal';

Toast Props

Prop Type Default Description
id (required) string Identifies individual toasts and differentiates them. If two toasts reuse the same id, the second one might not get displayed.
action shape
  • onAction - func (required)
  • label - string (required)
 Adds an action button to the toast
alertType enum
  • 'attention'
  • 'info'
  • 'success'
 'info' Gives a default left border and header with icon that corresponds to status level
headerText string Title text displayed at the top of the modal
manager shape
  • register - func
  • unregister - func
 ToastManager Manages which toast is shown at a given time
onDismiss func Called when a toast is dismissed either manually or automatically
onNeverShown func Called when a toast is "dismissed" without ever being shown
...rest Passthrough props are assigned to the current ToastTemplate

ToastTemplate Props

Prop Type Default Description
alertType (required) enum
  • 'attention'
  • 'info'
  • 'success'
 Sets the icon and border color
action shape
  • onAction - func (required)
  • label - string (required)
  • labelForScreenReader - string
 Custom action to be displayed under 'Dismiss'
headerText string Summarizes the message in the toast
onDismiss func Callback for toast dismissed
...rest Passthrough props to the root div element that wraps the ToastTemplate

ToastPortal Props

Prop Type Default Description
container Element document.body Containing element
...rest additional properties are not used
