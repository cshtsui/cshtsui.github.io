MultiField

MultiField displays a group of form inputs in a row along with corresponding labels, hint text, and error messages.

Usage

ReduxMultiField provides greater built-in functionality than MultiField, including features such as validation and state management. If you do not have strong implementation reasons for using MultiField, ReduxMultiField is recommended instead.

MultiField provides a responsive layout for a row of input fields. It's an extension of FormField and most of that component's guidance applies. The only difference is that MultiField displays multiple inputs in a row, each with a label above it. MultiField can also add an error message for each input and a single hint text message (the helper text below an input) for the whole group.

When to Use

MultiField places multiple inputs in a single row, but maintains separate labels and error messages for each input. This makes it useful for a group of inputs that are independent but related. It's not a good fit for inputs that depend on each other and don't make sense when separated. Examples of these include:

Intended for independent but related inputs:

  • First name, Middle name, Last name
  • City, State, ZIP code
  • Start date, End date

Not intended for dependent inputs:

  • Height, broken into two inputs: feet and inches
  • Weight, broken into two inputs: lb and oz
  • Social Security number, broken into three inputs with dashes between them

For dependent inputs, a better solution is to create a custom input component and use it in a FormField. See the Implementation section for details.

Design

MultiField is intended for inputs that are related and make sense to present together in a single row. Do not use it to group inputs that aren't directly related just to condense a form as this leads to a haphazard or confusing layout.

Find more guidance on the Form Design Foundations page.

Spacing and Layout

The recommended amount of spacing is already built into MultiField. The spacing can be customized; inputs should be kept close together, though, so that they visually appear as a group.

Avoid wrapping inputs. MultiField's purpose is to show related inputs on a single row.

Error Messages

MultiField supports separate error messages for each input. These messages are displayed below the row of inputs (and below the hint text, if any), aligned to the left. If multiple error messages are triggered, they are shown on separate lines, one per line, and in order of their corresponding inputs (left to right).

Implementation

MultiField is an extension of FormField that supports multiple inputs in the same row. The fields prop expects an array of objects, where each object is similar to the props supported by FormField. This is because the inputs behave as if each was in its own FormField. Each has its own labelText, value, onChange and so on. Apart from this, the API is similar to FormField. You can see which props are input-specific and which apply to the entire row in the API table.

If you want a dependent set of inputs with a shared value, errors, and callbacks (e.g., height divided into feet and inches), you can create a custom input component (see the Input component API section of the Forge Extension Generator Readme for more info).

Input widths

Fitting multiple inputs in a single row can require greater control over input widths. When determining the widths, keep two things in mind:

  1. Avoid letting the inputs wrap on any supported viewport. When the inputs wrap, all errors appear below the last input. This makes it hard for users to associate errors with any of the previous inputs.
  2. The width of the input determines how much space its label has before it wraps. For very small inputs, try to use short labels.

To set the width of an input (and its label) you can add a class with the className prop within fields and target it in CSS or set the inputWidth prop with any valid CSS width value.

Hint Text

MultiField only shows a single line of hint text (the helper text below an input), so all props related to hint text are at the top level, not in the fields array. Inputs can suggest what hint text to display, but if multiple inputs suggest hint text, only the most recent one will be displayed. Our focus here is on supporting the anticipated use case where an input suggests hint text only while it's focused.
