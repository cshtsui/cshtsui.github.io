Usage

ReduxFormField provides greater built-in functionality than FormField, including features like validation and state management. If you do not have strong implementation reasons for using FormField, ReduxFormField is recommended instead.

FormField is meant to be used as a child of Form.

FormField provides a responsive layout for inputs being used in forms. Additionally, FormField bundles a given input with a label, a component for error messages, and a component for informational (hint) text. FormField also provides styling for common cases such as indicating a field is required. Be sure to review these provided conveniences before deciding to construct a form manually.

Design

The advantage of FormField is it provides a recommended, out-of-the-box responsive layout. However, the following design points still need attention:

Group related form items

If a form asks about two different topics, split the form into separate sets of fields.

Use labels that are descriptive, concise, and not repetitive

Labels should describe the information required, without relying on context or memory. They should be short enough to fit without wrapping. For cleanliness, omit punctuation or any words that would appear in all labels. Move repetitive information out of labels and into a heading (e.g., Patient name, Patient address, and Patient phone can become Name, Address, and Phone in a section called "Patient Contact Information").

Label text should use sentence case ("Patient name", not "Patient Name").

Do not use an input's placeholder text instead of a label. This affects accessibility. Labels provide context. When users click into an input field, placeholder text disappears. This is not helpful for users with reading or memory issues who may need a persistent reminder of what the input field is for.

Present fields in a single-column layout

Multiple columns interrupt the vertical momentum of moving down a form. Rather than requiring users to reorient themselves, keep them in the flow with a single column and a separate row for each field.

Use logical sequencing

Stick to standard sequences for fields (such as City, State, ZIP Code) and arrange value choices in ascending or descending order for clarity. Also consider usage frequency, and list the most common values first. This guidance can supersede the single column guidance.

Explain any input or formatting requirements

If a field requires a specific format, type of input, or syntax, help the user understand what is needed. To avoid invalid entries, try setting the exact instructions as part of the FormField's hint text or show correctly-formatted placeholder examples in the input. Components that enforce format by default (e.g., DateInput) should be used for those data types whenever possible.

Provide specific error messages

Be as specific as possible about error messages. Relate what happened and suggest how to correct the problem.

Validation behavior

An application often has expectations about what information a user must enter and in what format. Validation tests those expectations, notifies the user of any errors, and blocks form submission until they are resolved. Inputs with an error are called invalid.

Required fields are the most common type of validation. Required inputs are invalid when empty. However, the error is not highlighted to the user until they interact with the field or try to submit the form, which helps to avoid error fatigue.

Find more guidance on the Form Design Foundations page.

Implementation

A FormField is a composition of a number of internal components:

  • An Input. FormField defaults to showing an Input if no component is specified with the inputAs prop.
  • A Label for the input.
  • An element to contain hint text.
  • An element to contain error messages.

Input Field Widths

Ideally, inputs in a form are only as wide as needed for the content being entered. This cannot be determined for all inputs, so use a mix of the following sizing strategies.

Default Width

The following input types are sized accordingly by default:

Input Type Default Input Width (Medium+ Breakpoint)
Input $input-medium-width (350px)
Checkbox $input-medium-width (350px)

In a Checkbox this translates to the max-width of the radio descriptions
DateInput The exact fixed-size required for a date entry of format 'MM-DD-YYYY' and the calendar icon - this width should never be altered
Textarea $input-large-width (500px)
Typeahead $input-medium-width (350px)
Multiselect $input-medium-width (350px)
RadioGroup $input-medium-width (350px)

In a RadioGroup this translates to the max-width of the radio descriptions
ReadOnlyInput Content-sized (width necessary to accommodate text)
Select Content-sized (width necessary to accommodate longest option)

On the small breakpoint, inputs will be full-width with labels shifted above the input and error messages always displayed below. The exceptions to this are content-dependent inputs (e.g., Select) or inputs where a size has been set explicitly.

Setting Input Widths Generally

When the exact character count of a piece of data is known, inputs should only be as large as needed to enter or display that data. For example, an input that takes two-character U.S. state abbreviations or zip codes should be sized accordingly. All input types can be given an exact width by using Input's' inputWidth prop which can be set from FormField in the following way:

<FormField
  labelText="I am an Input component with an inputWidth"
  inputWidth="200px"
/>

However, if the field is set wider than the container will allow, the input width is reduced to fit within its FormField (i.e., max-width is 100% of its parent).

Error Messages

Form error messages are typically the result of validation. Use ReduxFormField instead of FormField to take advantage of Forges built-in form validation. FormField allows explicitly set error messages with the error prop. Note that some inputs generate their own error messages based on user interaction (e.g., DateInput will generate an error if the date format is entered incorrectly). FormField will automatically display error messages generated by its input field if the error prop has not been set manually. Use the onErrorChange prop to handle the inputs error message change events.

Hint Text

Additional instruction for the FormField can be presented by using the hintText prop. Note that some inputs generate their own hint text based on user interaction (e.g., DateInput displays t-shortcut messages in hintText as they are entered). FormField will automatically display hint text generated by its input field if the hintText prop has not been set manually. Use the onHintTextChange to handle the input's hint text change events.

Passing Props to FormField Children

Any props not part of the FormField API are passed through to the wrapped input with the exception of className (see Styling FormField and its Children). This allows for setting props of the underlying input from the FormField level, and is also a convenience for those introducing custom components into their forms. It is up to the underlying input component to implement the usage of the passed through prop(s).

Styling FormField and its Children

Input widths can be set using dedicated props described in the Input Field Widths section.

Other localized styling of Forge forms in this manner is discouraged as a general practice. Doing so defeats the maintainability of user interface patterns and undermines the overall goal of providing consistency to athenaNet user interfaces. However, there may be instances (e.g., legacy integrations or prototyping) where its necessary to make some tweaks to the default Forge visuals.

Passing a class reference to the className prop in FormField applies styles to the root element of the entire field (i.e., the FormFieldLayout). To target a specific child of the FormField, use the class as a part of a CSS selector along with the FormField HTML API to help target the child needing adjustment. Make sure you have the appropriate level of specificity for targeting your chosen element. Forge components have at least two class selectors of specificity.

For example, the following makes the label of the field uppercase.

In your styles:

.fe_f_all .uppercase .fe_c_label {
    text-transform: uppercase;
}

In your component render code:

<FormField className="uppercase" labelText="Age" />

Again, these types of approaches are generally discouraged. Instead, consider working with user experience and/or the Forge team to arrive at consistent, reusable solutions.
