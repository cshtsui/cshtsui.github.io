Usage

Modal attracts attention to an issue or problem that must be immediately resolved before a workflow can proceed. It disables the main content with a darkened overlay until the user explicitly interacts with the dialog. Some typical use cases include:

  • Soliciting users for small amounts of additional detail without large workflow context changes that may prove confusing or onerous
  • Interrupting the user to communicate a potentially hazardous situation (e.g., patient safety)
  • Requiring users to confirm a destructive action (e.g., deletion)

Use modals sparingly because they interrupt a user's workflow. Excessive user interruptions with modals reduce their overall effectiveness in the application.

Modals are intended for short, alert-style interactions. To present a workflow in a pop-up dialog, use Lightbox instead.

Avoid having modals that open another modal dialog.

Design

The Modal component has several parts that require design consideration.

Modal Header

This is the title of the dialog and should indicate why the modal is appearing. Title text should use title case (e.g., "Edit Details"). This header also includes an icon allowing the dialog to be closed.

Body

This is the dialog content. For modals to be effective they need to be concise. Consider if a Modal is the correct pattern based on whether the following cases apply:

  • The form is long. Having a form that requires scrolling within a dialog indicates that the modal is likely not the correct container for that form.
  • The dialog contains multiple screens of information within a modal dialog. This is a sign that the task is too complex for a modal dialog.
  • The content scrolls. Vertical scrolling will occur if the content exceeds the area inside the modal; try to minimize content in the modal. Horizontal scrolling is not enabled within the modal.

Footer

Every Modal should have a clearly labeled call-to-action primary button. By design, these cannot be removed since the use of a Modal should be to convey unambiguous information or drive action. Button text should use title case (e.g., "Save Changes").

Overlay

Modal can be configured to be dismissed by clicking anywhere in the overlay (the darkened background). However, when displaying urgent information, do not allow users to dismiss the dialog by clicking the overlay, in order to avoid accidental dismissals.

Implementation

Modal does not close itself. Its parent is responsible for setting the show prop to false in the onHide and onPrimaryClick callback functions.

onPrimaryClick is called when the primary button is clicked. onHide is called for all other interactions that should close the modal (e.g., clicking on the overlay). Therefore, implementing the close functionality within onHide is required, unless disableClose is set to true.
