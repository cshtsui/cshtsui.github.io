Forge Version 2.0

Forge 2.0 is now available!

Forge 2.0 has been released to GA. It is no longer in beta.

How do I upgrade to Forge 2.0?

See the Confluence page on upgrading to Forge 2.0 for instructions.

What's different in 2.0?

Highlights of Forge 2.0 include:

  • Refactored and improved code
  • Tree shaking - Components can be imported individually to reduce your bundle size
  • Accessibility improvements
  • Upgraded extensions - All core extensions from 1.x are now fully part of Forge Core
  • New components - MultiField, ReduxMultiField, RemovableTag, StatusTag, and 3 new extensions (Toast, ToggleSwitch, and MultiselectAsync)
  • New guidelines on form design and interruption design

See the 2.0 Confluence page for full details and known breaking changes

Will Forge 1.x still be supported in the future? Will it be updated?

We will continue to support Forge 1.x with bugfixes until further notice. No new components or extensions will be built on the 1.x branch by the Forge team.

Also, we're no longer accepting extensions built on Forge 1.x. Please use 2.0 for extension development instead.

We do not recommend using Forge 1.x and 2.0 together on the same page.

Does Forge work on mobile?

The Forge team's current priority is building UI components for core athenaNet web pages, intended for use in our set of supported desktop browsers. We have not made specific concessions to use these components on mobile browsers, nor have we undertaken formal testing of the components on mobile browsers. However, while building components, the Forge team accounts for multiple viewport widths through a CSS media query based breakpoint system. Forge also supplies 2 components (GridCol and GridRow based on the Foundation Flex Grid) that can be used to make fluid layouts suitable for mobile web use.

Why are the Forge class names so long?

Forge adopts a structured and namespaced CSS class naming strategy to handle styling at enterprise-scale and styling that may be forced to live alongside legacy styles. The Forge naming strategy is based on a well-known architecture called BEM. The following rules are observed:

  • CSS namespace: all classes associated with Forge are prefixed with a global namespace: .fe_ to reduce the likelihood that Forge class names exist in stylesheets not deriving from Forge. For example, Bootstrap uses broad names like 'row' which if also used by Forge, would create styling conflicts.
  • Class prefixes: in addition to the .fe_ global namespace, Forge adds prefixes to each class to make it more apparent at-a-glance what job that class is doing. The categories of class prefixes are:
    • c_ for UI components, such as .fe_c_button
    • l_ for layout-related styles, such as .fe_l_grid-row__col
    • u_ for utilities, such as .fe_u_margin--top-large
    • is_ for specific states, such as .fe_is-disabled
    • f_ for features, such as fe_f_typography -- more about feature classes in the feature classes FAQ
  • Forge CSS names will otherwise follow BEM standards.

Learn more on the Forge HTML / CSS Guidelines Confluence page.

What are feature classes?

Feature classes are accelerators for applying certain Forge styles to elements in your application. Most Forge styles require specific Forge classes such as fe_c_button. However, putting a Forge-scoped class on every <p> in your application would be a lengthy process. The fe_f_typography class applies standard Forge styles to all the base typographical elements in your page, such as <p> and <a>.

fe_f_all is a catch-all feature class that incorporates the styles from all other feature classes. fe_f_all also must live at the root of the Forge part of your application, as Forge components are expecting the base typographical styles and normalize styles to be present. Forge components also utilize the fe_f_all class in their CSS selectors.

Does Forge have a set of common constants like colors?

Depending on what you need to do, you may be able to take advantage of some of Forge's typography utility classes or layout utility classes to get the standard styles you're looking for onto your components.

If you do need access to the Forge colors as variables (e.g. to output within your own classes for application-specific components), all of Forge's colors are represented within semantic Sass variable maps. (Sass is a CSS preprocessor that allows for things like variables and functions with CSS). The color page lists the syntax to access each color in that color's palette swatch, e.g. map-get($color-interaction, default).

For those getters to work, you will need to import all the Forge constants first in your main .scss file via:

import '@athena/forge/sass/forge-abstracts';

You may need to add 'node_modules' to the include path for your Sass compiler, depending on your setup.

If you started from forge-app, you should already have access to the Forge constants set up in your index.scss file.

Other common constants available in Forge as Sass maps include:

  • Font sizes
  • Font weights
  • Standard margin sizes
  • Standard padding sizes
  • Breakpoints

Additionally, the standard margin and padding constants are available in JavaScript from the JS constants file.

What is the Forge Workspace?

The Forge workspace is the development environment for Forge. It is based on storybook and offers certain developer-leaning features like knobs. It does not provide the design, usage, and implementation guidance that Forge Guide does, and should only be used in conjunction with Forge Guide.

Is Forge tied to Redux? Do I have to use Redux to use Forge?

In order to provide conveniences like validation in Forge forms, there needed to be a state management architecture in place. Redux is a very popular one both within and external to athena, and Redux Form is a vetted and popular add-on to Redux that focuses on form management. (Learn more about how Forge evaluates third-party libraries.) However, Form and FormField (as opposed to ReduxForm and ReduxFormField) exist specifically to provide an opt-out option from the Redux / Redux Form architecture. These components are actually the styling / layout building blocks for their Redux counterparts, and are agnostic to the state management architecture you choose to adopt.

General / Product

Which browsers does Forge officially support?

Forge supports the same browsers that athenaNet officially supports, namely:

  • Chrome (Windows 7, Windows 10, + Mac)
  • IE11 (Windows 7 + Windows 10)
  • Safari (Mac)

Can I use Forge in my product?

Forge components are built using the React JavaScript framework. Please consult with your product's technical leaders to understand readiness and architectural best practices for incorporating this new technology. The recommended practices for integrating React may differ by product.

Existing products / workflows

We are not recommending that product teams go back and redesign all of their workflows in one push. However, as workflows are redesigned and as net new features are added to your existing products, you can use Forge. The transition to the look and feel will be intentionally gradual.

New products / workflows

Using Forge is recommended as it should expedite your design and build process and, if applied correctly, will ensure higher usability.
