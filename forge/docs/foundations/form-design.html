Form Design

Forge form design seeks to balance usability, aesthetics, and information display density for enterprise software users.

This page documents the general Forge design direction for forms. Where practical, Forge form components have implemented this guidance for convenience of design and development. The Forge form components are solid starting points and are recommended for most projects involving basic CRUD (create, read, update, delete) functionality (e.g., system administration screens, forms powering report generation, short data entry forms).

For cases involving exceptionally large numbers of form fields, highly optimized data entry needs, or legacy compatibility requirements, it may be necessary to vary from the implemented components. In these cases, carefully considering the following design, interaction, and usability decisions will help create a user experience compatible with the overall Forge design system.

General Design Guidelines

A form displays a set of related user input fields in a structured way. When designing a form, keep these general guidelines in mind:

  • Minimize the size of forms and make sure the inputs are really needed.
  • Minimize effort for data entry. For example, build in auto capitalization and auto formatting for dates and phone numbers.
  • Provide clear labels and instructions to minimize input error.
  • Group related inputs into sections and distinguish them visually using typography or graphics.

In general, long forms should be avoided. One exception is data entry workflows where the user simply types in lots of information. A single long form is appropriate for this and makes input easier. A long form is not a good fit for workflows involving complex information or questions, multiple steps, or logic branches where inputs can change depending on what the user entered earlier in the form.

Introductory text can be included at the top of the form to set context and provide general instruction. However, it doesn’t need to describe what is obvious from looking at the form (e.g., “Complete all required fields and click Continue”). See the Content Style Guide for more information about writing UI copy.

Input Fields

A standard form has one user input field per row.

Single input layouts can be customized to have multiple dependent inputs under a single label. These are inputs that are parts of a whole and don’t make sense when separate. Examples of this are:

  • Social Security number, broken into three inputs
  • Height, broken into two inputs: feet and inches
  • Phone number, broken into multiple inputs for area code and number

A group of inputs that are independent but closely related can be displayed in a single row with labels on top to make them visually connected. Examples of this include:

  • First name, Middle name, Last name
  • Date ranges
  • City, State, ZIP code

A form can contain both single input and multiple input fields. When possible, align left edges of all input fields to establish visual order.

Labels should be nouns (“First name”) rather than instructions (“Enter first name”). Use sentence case capitalization (“Patient first name”, not “Patient First Name”). Do not put a colon (:) at the end of the label.

For accessibility reasons, do not use placeholder text instead of a label. Labels are necessary for screen readers. They also provide consistent context. Placeholder text deliberately disappears, which removes that context.

Label Placement

Forge supports both top label and side label for form.

Top Label

Top label is the default design for forms with multiple inputs in a row (see exceptions in “Side Label”). Research shows "Top label" enables quick label/form field pairing and reduces time to complete form. It has added benefits of allowing for long labels, localization, and mobile compatibility.

Side Label

Side label is the default design for forms with a single input per row.

Side label is also used for cases where multiple related inputs are in a string or sentence format.

The form field designs are responsive. A side label on a single input layout form will change to as top label at small breakpoints, including mobile scenarios.

Side labels should be left aligned and wrap when the string is too long. Minimize the gap between the label and the input. Use the labelWidth prop in FormField to adjust the amount of space allotted to the label.

Do not mix top and side label input fields in a form.

Action Button

Action buttons (e.g., Submit, Confirm, Next, Cancel) should be placed at the bottom of the form. Single or multiple action buttons can be used. When multiple buttons are used, the primary action button should be placed on the right. Action buttons should be aligned with the left edge of input fields. When there are errors in the form, clicking the action buttons will take users to the first input field with an error. Use title case for button text (e.g., “Save Form”, “Submit Data”).

Layout

Basic Structure

A basic form should use single column layout by default.

Research shows that a linear single column form is easier to visually traverse and faster to complete. It is best suited for an interface that requires heavy data entry. The single column structure is also the most usable form structure for mobile interfaces so aligning on this layout has the advantage of reducing effort to create interfaces that simultaneously address desktop and mobile users.

Single column form can have multiple related fields displayed in a row (see Input fields).

Two Column Forms

In a desktop application, a two-column layout can be used in exceptional cases, such as:

  • Many input fields (>25) all need to be on one screen
  • Completion or review of the form requires comparing or referring back to information
  • Input fields can be grouped under a header, and sequential order of the groups is not critical

Overall, use single column layout for forms whenever possible with single or multiple fields per row.

Spacing

Forge supports two vertical spacing designs for forms to cover the needs of both client- and consumer-facing products.

Default

Spacing between FormField rows is 20 pixels. Use default spacing for forms when possible.

Compact

Spacing between FormField rows is 16 pixels. Use compact spacing only when it is important to display a large quantity of form fields on one screen (for example, in a data entry workflow).

For multiple inputs in a single row, vertical spacing between input fields is 8 pixels.

Input Field Width

In general, making the input fields width match the expected input is important to the usability of the form.

When the character count is unknown, let the inputs use their default widths. For most inputs, this is 350px, but there are some exceptions. A Textarea is wider at 500px and select widths are determined by the longest piece of content. If the form itself is narrow, the width of the form constrains the width of its inputs.

For multiple inputs in a row with labels on top, the input field should be equal to or wider than the form label. The recommended minimum width of an input field is 48 pixels. When setting specific widths for inputs, consider how it will look on different screen sizes. Avoid letting the inputs wrap on any supported screen size.

Validation

Required Field

Required input fields are indicated with a vertical blue bar and the text “ - Required” in the corresponding label.

Inline Alert

Forge supports alert placement for both top label and side label forms. This guidance applies to both top and side label forms.

  • Form with a single input field per row: Alert will be placed either under the form field or to the right of it, depending on available space.
  • Form with multiple inputs in a row: Alert will be placed underneath the fields. When multiple alerts are triggered, they should be stacked by their order within the row (left to right).
