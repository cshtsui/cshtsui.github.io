Accessibility

The Forge design system is built to meet WCAG 2.0 AA accessibility requirements. Using this system is just the first step; you still need to take action on your part to build accessible products.

Types of Impairments

Technology and assistive tools (such as screen readers) help make the web usable to people with disabilities and other impairments. But these tools can't do all the work just on their own. For these tools to provide a good experience to their users, applications need to be created using specific accessibility practices.

The Forge team builds accessibility into its design system to address 3 types of impairment or disability: visual, motor, and cognitive. These cover a number of conditions and situations and affect different areas of the user experience (see table below).

Type of Impairment Examples of Conditions and Situations Examples of UX Impact
Visual
  • Blindness
  • Partial sight loss (from age or illness)
  • Color blindness
  • Low-quality monitor
  • Unable to see interface
  • Unable to perceive color differences among elements
Motor
  • Muscle tremor
  • Musculoskeletal disorders
  • Carpal tunnel syndrome
  • Hands partially full
  • Trouble clicking on small targets
  • Trouble using mouse
Cognitive
  • Memory problems
  • ADHD
  • Dyslexia, dyscalculia
  • Distracting environment
  • Multitasking
  • Speaking English as a second language
  • Trouble remembering where you are in a multi-step process
  • Trouble reading long chunks of text

General accessibility practices usually include 2 more types: auditory (which can affect how users interpret sound) and vestibular disorders and seizures (which can affect how users react to animation). The Forge design system will address these when there is demonstrated product demand.

Why This Matters

1. Lack of accessibility can lead to legal problems.

  • Inaccessible websites' gaps in usability may be considered discrimination. One way that people have approached this problem is through legal action.
  • This isn't limited to government web technology, which is required by law to be accessible (specifically, Section 508 of the Rehabilitation Act). Epic was sued in January 2019 for the lack of accessibility in its EHR software.

2. Anyone may need these features temporarily or in certain situations.

  • It's not just people with ongoing conditions who benefit from accessibility features. All of us may have a temporary issue or a situation that limits how well we can use technology. For example:
    • Having a broken arm that makes it hard to use a mouse
    • Using a mobile phone with one hand while holding an infant with the other
    • Being stuck with a low-quality monitor that renders colors and visuals poorly
    • Working in a high-distraction environment such as an ER front desk (which can tax users' cognitive abilities)

3. More people may benefit from these features than just the target users.

  • A feature built to help users with a specific need or disability can end up helping other users, too. For example:
    • Designing for keyboard-only use helps both people who can't use a mouse and data entry power users who save time by not switching between mouse and keyboard.

4. More people can use our products when they're accessible.

  • We want a wide range of people to use our products and to enjoy using them. When we build for accessibility, we welcome more possible users. By meeting them where they are, we tell them that they matter to us.
  • In the healthcare arena, people with disabilities face challenges both as providers and patients. By accounting for accessibility in our products, we're helping to address these problems.
    • Doctors With Disabilities: Why They’re Important (New York Times)
    • Disability and Access to Healthcare (CDC)

Forge's Support for Accessibility

Accessibility Standards

The Forge team builds accessibility features to meet the Web Content Accessibility Guidelines (WCAG) 2.0 level AA.

Interaction Styles and Assistive Technology

Forge components are built to be functional when used with keyboard only (no mouse) and when interpreted by screen readers.

For example, the Toast component provides a keyboard shortcut to dismiss it with keystrokes rather than with a mouse. It is also built to be announced by screen readers when it appears; otherwise, users with visual impairments might not know that it is present.

Color

The Forge color palette has been designed to meet usability criteria and accessibility requirements (it's not just a matter of choosing which colors "look good").

Contrast: WCAG 2.0 AA requires specific contrast ratios for text and background colors. The Forge color palette supports this, when used appropriately. Forge components are also designed to meet this contrast level; this is confirmed with testing (see Testing Tools and Checklists).

Colorblind support: Forge components incorporate color but don't rely on it solely to communicate. For example, the Banner uses alert colors and icons to convey level of importance. The Forge team has consulted people with color blindness in testing color choices.

Copywriting

Writing clear and simple copy is an important part of accessibility. Product teams are responsible for writing their own copy. Forge provides support for this in the content style guide and in component-specific guidance.

Code

Under the hood, Forge's components use semantically correct HTML markup along with attributes from the Web Accessibility Initiative - Accessible Rich Internet Applications (WAI-ARIA) standards where possible. These provide information to help assistive technology interpret the screen's hierarchy and elements.

Forge's developers use eslint-plugin-jsx-a11y, an eslint plugin that spots accessibility issues during coding. For example, if an element has an added onClick behavior but no equivalent for keyboard use, the plugin will call out this issue for attention.

Forge's Testing Process

The Forge team uses these tools and processes to carry out accessibility testing as part of component development:

  • Screen reader testing
    • Apple Voiceover (system utility) with Safari
    • NVDA screen reader with Firefox
    • The process includes confirming that:
      • Page elements are read aloud in the expected order (the same order as they appear visually)
      • Alt text is present where needed and is read aloud
      • Dynamic elements (such as Toast) are properly announced
  • Keyboard-only use
    • No special software is needed for testing keyboard-only use
    • The process includes confirming that:
      • The screen can be navigated using the tab key
      • Elements get tabbed to in the expected order (the same order as they appear visually)
      • Users can't navigate to hidden elements
      • Focus styles are visible
      • Users can actually interact with elements, not just navigate to them
      • Hotkeys/shortcut keys work

Making Accessible Products with Forge

The Forge design system is steadily working to include accessibility support for all components in the main Forge package, but this doesn't mean that any product using it is automatically accessible. Teams must also take steps themselves during design, development, and testing processes.

Design

Design aspects to consider include:

  • Workflow design (e.g., planning for keyboard-only use and screen reader use, assessing the workflow for parts that may be cognitively challenging)
  • Color choice
  • Copywriting

WebAIM's Web Accessibility for Designers offers a list of principles to follow.

Development

Developers should review the documentation for Forge components. Components may have accessibility features that require implementation. For example, some include props for copy used specifically by screen readers to provide additional context.

For the best results, test keyboard navigation and screen reader compatibility often, not just at the end of development. If you are using React and eslint, consider adding the eslint-plugin-jsx-a11y plugin to your project.

Testing

Teams are responsible for testing their own applications for accessibility. This can include:

  • Testing keyboard-only and screen reader use
  • Assessing cognitive accessibility
  • Confirming color contrast ratios
  • Checking how the application looks on a lower-quality monitor or projector

See the list of testing tools below for useful plugins, checklists, and apps for accessibility testing.

To thoroughly test and review your entire application, consider having an accessibility audit done by experts (which can include external consultants). This will find issues in workflows and the overall application beyond just the individual Forge components.

Forge Extensions and Community Contributions

WCAG 2.0 AA accessibility compliance is now required for Forge core extensions (components that will eventually be incorporated into Forge core library). The Forge team will not merge new component code to the core library unless it is accessible.

Forge community extensions (components developed by product teams, often for specific product use) are supported by the teams that build them; they can meet any level of accessibility that their team is able to support. Keep in mind that a mix of accessibility standards in a screen will lower the overall accessibility experience.

See the Extensions page for more detail about core and community extensions.

Reporting Accessibility Issues in Forge

If you spot any areas in Forge where accessibility support is missing or could use improvement, please report them to the Forge team through the Issue Tracker.
